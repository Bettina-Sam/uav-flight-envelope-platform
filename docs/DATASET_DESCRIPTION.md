# Dataset Description

File: `backend/data/uav_synthetic_dataset.csv` (6,000 rows, generated by
`app/dataset_generator.py`, physics ground truth from `app/physics.py`).

Each row = one randomly sampled UAV design configuration + its computed flight-envelope
outputs. **Not** one row per altitude — altitude is swept internally by the physics
engine to derive the envelope, then the representative performance at the
recommended altitude is recorded as the row's output values.

## Input features (14) — sampled uniformly within these bounds

| Column | Bounds | Meaning |
|---|---|---|
| `mass_kg` | 7.0 – 25.0 | Total UAV mass (empty + payload) |
| `payload_kg` | 0.0 – min(5.0, 0.4×mass) | Payload mass, constrained to ≤ 40% of total mass |
| `wing_area_m2` | 0.30 – 0.95 | Wing planform area (S) |
| `wingspan_m` | 2.0 – 4.2 | Wingspan (b) |
| `cl_max` | 1.1 – 1.6 | Maximum usable lift coefficient |
| `cd0` | 0.020 – 0.040 | Zero-lift (parasite) drag coefficient |
| `oswald_efficiency` | 0.70 – 0.88 | Span efficiency factor (e) |
| `battery_wh` | 500 – 2000 | Usable battery pack energy (Wh) |
| `motor_max_power_w` | 350 – 1300 | Peak motor electrical power **per engine** |
| `num_engines` | {1, 2, 2, 2, 2, 2, 2, 3} (discrete, weighted) | Number of engines. Platform baseline is **twin-engine (2)**; 1 and 3 included in small proportion so the ML model has some signal for engine-count effects |
| `propulsion_efficiency` | 0.60 – 0.85 | Baseline (sea-level) propeller/propulsion efficiency (unitless) |
| `cruise_speed_ms` | 14.0 – 30.0 | Fixed design cruise true airspeed |
| `aspect_ratio` | *derived* | b² / S |
| `wing_loading_kg_m2` | *derived* | mass / S (in kg/m² for readability) |
| `power_loading_w_kg` | *derived* | (motor_max_power_w × num_engines) / mass — uses TOTAL power across all engines |

## Target outputs (13)

| Column | Meaning |
|---|---|
| `min_altitude_m` | Regulatory/practical operating floor (fixed at 30 m for this class) |
| `max_altitude_m` | Highest altitude with positive rate of climb and V ≥ V_stall |
| `recommended_altitude_m` | Engineering-optimal altitude — see `docs/FORMULA_SHEET.md` |
| `service_ceiling_m` | Altitude where ROC drops to 100 ft/min (0.508 m/s), interpolated |
| `absolute_ceiling_m` | Altitude where ROC = 0, interpolated |
| `rate_of_climb_ms` | ROC at the recommended altitude |
| `range_km` | Still-air range at the recommended altitude |
| `endurance_hr` | Flight endurance at the recommended altitude (20% battery reserve) |
| `power_required_w` | Aerodynamic power required at the recommended altitude |
| `lift_n` | Lift force at the recommended altitude (≈ weight, by construction) |
| `drag_n` | Drag force at the recommended altitude |
| `l_over_d` | Lift-to-drag ratio at the recommended altitude |
| `safety_status` | 0 = SAFE, 1 = CAUTION, 2 = CRITICAL (rule-based classification) |

## Why these bounds

They correspond to a 7-25 kg class electric fixed-wing mini-UAV — comparable in scale to
published Indian and international ISR mini-UAV designs (see the project's baseline
reference table in the original scoping conversation). They are **not measured from a
real fleet** — this must be stated clearly in any report or viva: this is an
engineering-judgment sampling range chosen to keep the dataset physically realistic,
not a fitted distribution from real aircraft data.

## Regenerating with different bounds

Edit `BOUNDS` in `backend/app/dataset_generator.py`, then re-run:
```bash
python -m app.dataset_generator --n 6000 --seed 42
python -m app.train_model
```

## Using real flight-test data instead of (or alongside) synthetic data

`backend/app/data_loader.py` provides a `load_training_data(source=...)` abstraction with
three modes:

- `synthetic` (default) — the physics-generated dataset described above.
- `real` — a real dataset at `backend/data/real_uav_flight_data.csv`, which **must have
  exactly the same columns** as listed in this document. A blank template with the
  correct header row is provided at `backend/data/real_uav_flight_data_template.csv`.
- `blended` — synthetic + real combined, with the real rows oversampled so a small real
  dataset can still meaningfully influence training against a much larger synthetic set
  (a standard technique when real data is scarce relative to simulated data; see
  `data_loader.py` docstring for the exact oversampling method).

Switch source with the training CLI flag:
```bash
python -m app.train_model --data-source real
python -m app.train_model --data-source blended
```
No changes to the physics engine, FastAPI backend, or frontend are needed when real data
becomes available — only the training data source changes. This is the single biggest
disclosed limitation of the current model (see `docs/ML_METHODOLOGY.md`), and this is the
seam where it gets closed.
